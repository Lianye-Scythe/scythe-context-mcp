# Release Process

This project publishes through GitHub Actions after a `vX.Y.Z` tag is pushed. The publish workflow verifies the package, skips npm publishing if that exact version already exists, and creates a GitHub Release from `CHANGELOG.md`.

## Preconditions

- Work from `main`.
- The worktree must be clean.
- GitHub Actions must have `NPM_TOKEN` configured for npm publish.
- Do not include `.env`, `.scythe-context/`, `.repo-beacon/`, `local/`, API keys, proxy tokens, or private reference files.

## Prepare a Patch Release

```bash
git checkout main
git pull origin main
npm run release:prepare -- patch
```

The release prepare script updates:

- `package.json`
- `package-lock.json`
- `src/cli.ts`
- `CHANGELOG.md`

Review the generated changelog section before committing. It should explain behavior changes from a user or maintainer perspective, not only list implementation details.

## Verify Locally

Run the standard release checks:

```bash
npm run verify
npm run bench:gate
```

When API usage is acceptable and `GEMINI_API_KEY` is configured, also run:

```bash
npm run bench:gate:hybrid
```

`npm run verify` includes tests, build, CLI smoke, production audit, and `npm pack --dry-run`. If a tarball is left in the repo root after local checks, remove it before committing.

## Commit, Tag, and Push

```bash
git add package.json package-lock.json src/cli.ts CHANGELOG.md
git commit -m "Release X.Y.Z"
git push origin main
git tag vX.Y.Z
git push origin vX.Y.Z
```

Pushing `main` runs CI. Pushing the tag runs the publish workflow.

## Confirm Publication

```bash
gh run list --limit 5
gh release view vX.Y.Z
npm view scythe-context-mcp version
```

The expected result:

- CI succeeds on `main`.
- Publish succeeds on the tag.
- GitHub Release exists for `vX.Y.Z`.
- npm reports the new version.

## Reruns

The publish workflow is intentionally rerun-safe:

- If `scythe-context-mcp@X.Y.Z` already exists on npm, npm publish is skipped.
- If the GitHub Release already exists, release creation is skipped.
- The tag version must still match `package.json`.

## Release Notes

GitHub Release notes are generated by:

```bash
node scripts/release-notes.mjs X.Y.Z
```

The script extracts the matching `CHANGELOG.md` section and promotes `###` headings to `##` headings for GitHub Release formatting.
